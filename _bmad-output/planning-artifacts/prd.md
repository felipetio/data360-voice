# Product Requirements Document - Data360 Voice

**Author:** Felipe
**Date:** 2026-03-23

## Executive Summary

Climate disinformation spreads faster than the facts that could counter it. The World Bank maintains one of the richest climate datasets on the planet (300M+ data points, 200+ economies, 10,000+ indicators), but accessing it requires navigating complex APIs, databases, and taxonomies that most journalists, researchers, and citizens will never touch. The result: disinformation goes unchallenged because the official data to refute it is too hard to reach.

Data360 Voice is a conversational AI tool that lets anyone query World Bank climate data in natural language and receive verified, citable answers with complete source attribution. Ask "How has drought increased in Brazil in the last decade?" and get a data-grounded narrative with World Bank source citations, not a dashboard or a CSV.

Built for the Data 360 Global Challenge (Media Party + World Bank, category Data Dialogue), Data360 Voice serves three user groups: climate journalists who need sourced data on deadline, policy analysts preparing cross-country comparisons, and citizens seeking to understand climate impact without technical skills.

### What Makes This Special

Every claim is citable by design. DATA_SOURCE attribution is extracted directly from the World Bank API response, not generated by the LLM. The AI interprets and narrates, but cannot fabricate the underlying data or its provenance. In a landscape where AI hallucination is an active credibility crisis in journalism, this structural guarantee, not a feature toggle, makes Data360 Voice a tool that newsrooms can actually trust.

This is a trust intervention, not just a UX improvement. A transparent "I don't have data for that" is more valuable than a confident wrong answer. The convergence of the Data360 API's vector search, MCP for clean tool integration, and the peak of climate disinformation makes this the right product at the right time.

## User Journeys

### Journey 1: Ana, the Climate Journalist (Primary - Success Path)

Ana is a mid-career environmental journalist at a major Brazilian outlet. It's Tuesday afternoon, and her editor just asked for a piece on drought trends in Brazil for tomorrow's edition. She needs hard numbers with sources she can cite.

**Opening Scene:** Ana has 3 hours to file. She's used to spending most of that time hunting through INPE dashboards, cross-referencing World Bank spreadsheets, and hoping the numbers she finds are current. She often settles for the easiest number to find, not the most accurate.

**Rising Action:** Ana opens Data360 Voice and types: "How has drought increased in Brazil in the last decade?" The system searches the World Bank Data360 API, finds the relevant indicators, and returns a narrative response: drought severity increased 23% between 2014 and 2024 compared to a 12% global average. The response includes the exact indicator code, DATA_SOURCE attribution, and the most recent data year.

Ana follows up: "How does this compare to other South American countries?" Data360 Voice generates a comparative analysis, each country's data point carrying its own source citation.

**Climax:** Ana copies the response directly into her draft. Every number has a formal citation to the World Bank Data360 platform. Her editor can verify any claim by following the source. She didn't need to open a single spreadsheet.

**Resolution:** The article publishes with four cited data points, all sourced to the World Bank. Ana's piece is the most data-rigorous in her section. She bookmarks Data360 Voice. Total time from question to sourced data: 4 minutes.

### Journey 2: Kofi, the Policy Analyst (Primary - Alternative Goal)

Kofi works at an African think tank preparing a policy brief on CO2 emissions across the continent for an upcoming AU summit. He needs comparative data for 54 countries with methodology transparency.

**Opening Scene:** Kofi normally spends two full days extracting data from the World Bank API using Python scripts, then formatting it into tables. He's technically capable but the process is tedious and error-prone when handling dozens of countries.

**Rising Action:** Kofi types: "Show me CO2 emissions per capita for all African countries." Data360 Voice returns a structured response covering available countries with per-capita figures, source attribution for each data point, and the most recent year available. Some countries have data from 2022, others from 2020. The system flags the discrepancies transparently.

Kofi follows up: "Which African countries reduced emissions the most since 2010?" The system identifies trend data across the continent, noting where data gaps exist and which countries lack sufficient historical data for trend analysis.

**Climax:** Kofi gets a comprehensive, sourced dataset in 15 minutes that would have taken him two days of scripting. Each data point carries its source and recency, so his policy brief's methodology section writes itself.

**Resolution:** The policy brief goes to the AU summit with 30+ cited data points. Kofi's director asks how he produced it so fast.

### Journey 3: Clara, the Student Activist (Primary - Guided Experience)

Clara is a 22-year-old environmental science student preparing a presentation for a campus climate rally ahead of COP. She's passionate but not data-literate, and she's been burned before by sharing statistics that turned out to be outdated or misleading.

**Opening Scene:** Clara is building her slides and wants to make a strong case about climate impact in her home country, Colombia. She googled some numbers but isn't sure which sources to trust. A friend who's a journalism student told her about Data360 Voice.

**Rising Action:** Clara types: "What's happening with climate change in Colombia?" The query is broad, but Data360 Voice handles it by identifying the most relevant climate indicators (CO2 emissions, temperature anomalies, forest area change) and presenting a multi-faceted overview. The tone is clear and accessible, each claim backed by a World Bank source.

Clara gets curious: "Is Colombia doing better or worse than the global average on renewable energy?" Data360 Voice compares Colombia's renewable energy consumption to the world average, providing context about what the numbers mean.

**Climax:** Clara realizes she can trust these numbers because every single one has a formal source. She's never had that confidence before. She asks one more: "What about deforestation in the Amazon region?" Data360 Voice responds transparently that it only has country-level data, not regional, and shows Colombia's national forest area trends instead.

**Resolution:** Clara's presentation has six slides with World Bank-sourced data. When a professor challenges one of her numbers during Q&A, she pulls up the Data360 Voice conversation and shows the exact source. The professor is impressed.

### Journey 4: The Fact-Checker (Primary - Verification Flow)

It's election season. A viral tweet claims "Brazil's deforestation dropped 50% since 2020." A fact-checking desk at a news organization needs to verify this fast.

**Opening Scene:** The fact-checker has 30 minutes before the claim spreads further. Traditional verification would mean finding the right World Bank indicator, pulling the data for 2020 and the latest year, calculating the percentage change, and writing up the verdict. That's at least an hour of work.

**Rising Action:** The fact-checker pastes the claim into Data360 Voice: "Verify: Brazil's deforestation dropped 50% since 2020." The system identifies the relevant forest area indicators, pulls Brazil's data for 2020 through the most recent year, calculates the actual change, and compares it to the claimed 50% drop.

**Climax:** Data360 Voice returns a clear verdict: "According to World Bank data, Brazil's forest area changed from X to Y km² between 2020 and 2023, representing a Z% change. The claim of a 50% drop in deforestation is not supported by the available indicators." The response includes the exact DATA_SOURCE, indicator code, and data years. The fact-checker has a publishable verdict in under 2 minutes.

**Resolution:** The fact-check publishes within the hour, citing World Bank Data360 as the authoritative source. The claim is debunked with official data before it gains more traction.

### Journey 5: Edge Case - No Data Available

A journalist asks: "What are the PM2.5 air pollution levels in rural villages in northern Nigeria?"

**Opening Scene:** The query is specific and reasonable, but it requires sub-national granularity that the World Bank Data360 API doesn't provide.

**Rising Action:** Data360 Voice searches for relevant indicators. It finds country-level air pollution data for Nigeria but nothing at the sub-national or village level.

**Climax:** Instead of guessing or hallucinating, the system responds transparently: "I found country-level air quality data for Nigeria (PM2.5 annual mean: X µg/m³, source: WB_WDI_EN_ATM_PM25_MC_M3, 2022), but I don't have sub-national data for specific regions or villages. The World Bank Data360 platform provides country-level data only."

**Resolution:** The journalist gets what's available (national-level data with source) and a clear explanation of the limitation. No time wasted, no false confidence.

## Domain-Specific Requirements

### LLM Grounding Boundary

The system's trust proposition depends on a clear boundary between data interpretation and hallucination:

**Allowed (interpret):**
- Narrate API data: describe values, percentages, changes over time
- Compare data points: country vs. country, country vs. global average
- Describe trends: rising, falling, stable, accelerating
- Flag data gaps: missing years, unavailable indicators, stale data

**Forbidden (hallucinate):**
- Causal explanations: never explain *why* a trend exists unless the API data contains that context
- Predictions: never extrapolate or forecast future values
- External knowledge: never add facts, events, or context from outside the API response
- Editorial judgment: never characterize data as alarming, positive, failing, etc.

**When users ask "why?":** The system responds that it can report what the World Bank indicators show but cannot explain causation beyond what the data contains.

### Data Freshness

World Bank indicators are typically 1-3 years behind current year. The system must display the most recent data year on every response and warn explicitly when data is older than 2 years. This is inherent to institutional data, not a system limitation.

## Innovation & Novel Patterns

### Detected Innovation Areas

**1. Structural citation guarantee (not feature-level):** Most AI tools treat source attribution as a feature that can be toggled on or off. Data360 Voice makes it architectural: DATA_SOURCE fields are passed through from the API response, not generated by the LLM. The AI cannot produce a data claim without its source. This is a fundamentally different approach to the AI hallucination problem in data journalism.

**2. Conversational interface as trust intervention:** The innovation isn't "chatbot for data" (that exists). It's reframing conversational access as a mechanism for restoring trust in information. The Data 360 Global Challenge exists because of a trust crisis. Data360 Voice addresses it by making verification frictionless, not just access easier.

**3. MCP as the integration pattern:** Using Model Context Protocol to wrap an institutional API is an emerging pattern (2026). The MCP server is transport-agnostic and reusable: it works in Claude Desktop via stdio for development and in the web app via HTTP Streamable for production. This positions the MCP server itself as an open-source contribution post-challenge.

### Validation Approach

- **Citation accuracy:** Automated test suite verifying every data response includes DATA_SOURCE from the API (not LLM-generated)
- **Grounding boundary:** Test suite with queries that should trigger "I can't explain causation" responses
- **User validation:** At least one journalist (Gustavo's network) completes a real task with the prototype
- **Challenge validation:** EOI selection and finalist status as external validation of the concept

## Web Application Specific Requirements

### Project-Type Overview

Data360 Voice is a single-page web application built on Chainlit (mounted in FastAPI), delivering a real-time conversational interface via WebSocket (Socket.IO). The architecture is a monolithic ASGI deployment: one container running Chainlit for the chat UI and FastAPI for auxiliary endpoints, with PostgreSQL for persistence and caching.

### Technical Architecture Considerations

**Browser Support:**
- Modern browsers only: Chrome, Firefox, Safari, Edge (latest 2 versions)
- No IE11 or legacy browser support
- WebSocket support required (all target browsers support this)

**Real-Time Communication:**
- Chainlit uses Socket.IO for WebSocket connections
- LLM responses stream token-by-token to the UI
- Tool call status (searching indicators, fetching data) displayed as intermediate steps

**Responsive Design:**
- Desktop-first for MVP (primary users are journalists and analysts at workstations)
- Mobile-responsive layout via Chainlit's built-in responsive design
- No native mobile app planned

**SEO Strategy:**
- Not applicable for MVP (authenticated chat interface)
- Post-MVP: shareable conversation links may benefit from server-side rendering of answer cards

**Accessibility:**
- Best-effort for MVP, leveraging Chainlit's built-in accessibility features
- Semantic HTML and keyboard navigation where Chainlit supports it
- No specific WCAG level target for prototype

### Performance Targets

| Metric | Target |
|--------|--------|
| Time to first token (streaming) | <3s uncached, <1s cached |
| Full response completion | <15s uncached, <5s cached |
| Chat UI load time | <2s on modern broadband |
| Concurrent users (prototype) | 10-50 simultaneous sessions |

### Implementation Considerations

**Chainlit Mount Pattern:**
- Chainlit mounted as FastAPI sub-application via `mount_chainlit`
- Single ASGI process (Uvicorn), single deployment container
- Shared database connection pool between FastAPI and Chainlit

**MCP Client Integration:**
- Chainlit acts as MCP client with native handlers (`@cl.on_mcp_connect`, `@cl.on_mcp_disconnect`)
- MCP server runs as separate process, communicates via HTTP Streamable in production
- Tool calls rendered natively in Chainlit's chat step UI

**Deployment:**
- Docker container on Railway or Render
- Managed PostgreSQL with pgvector extension
- Environment variables for Claude API key, database connection
- WebSocket support required from hosting platform (sticky sessions for scaling)

## Non-Functional Requirements

### Performance

Performance targets are defined in the Web Application Specific Requirements section. Key NFRs:

- NFR1: Streaming responses must deliver first token within 3 seconds (uncached) or 1 second (cached)
- NFR2: Full response completion must occur within 15 seconds (uncached) or 5 seconds (cached)
- NFR3: The system must support 10-50 concurrent user sessions without degradation
- NFR4: World Bank API response caching must reduce repeat query latency to <100ms

### Security

- NFR5: Claude API key must be stored as environment variable, never in source code or client-side assets
- NFR6: Database connection credentials must be stored as environment variables
- NFR7: All external API communication (Claude API, Data360 API) must use HTTPS
- NFR8: No user PII is collected or stored beyond conversation content

### Integration

- NFR9: The system must handle Data360 API unavailability gracefully with clear user messaging
- NFR10: The system must handle Claude API rate limits with exponential backoff
- NFR11: Cached API responses must have configurable TTL (default: 24 hours for data, indefinite for metadata)
- NFR12: The MCP server must be transport-agnostic, supporting both stdio and HTTP Streamable without code changes to tool logic

### Offline Indicator Discovery

- FR37: The MCP server can return a curated list of popular climate and development indicators without any API call
- FR38: The MCP server can search local indicator metadata offline using relevance-scored substring matching
- FR39: Offline search results include relevance scores so the LLM can prioritize the best matches
- FR40: Indicator metadata and popular indicator data are loaded once and cached in memory for the server lifetime

### Temporal Coverage

- FR41: The MCP server can check the temporal coverage (start year, end year, available years) for a given indicator and database
- FR42: Temporal coverage extraction uses the existing metadata endpoint with OData filtering
- FR43: The MCP server enforces a 3-step data retrieval workflow: search indicators → check temporal coverage → retrieve data

### MCP Prompts & Resources

- FR44: The MCP server provides a `compare_countries` prompt that guides multi-country indicator comparison
- FR45: The MCP server provides a `country_profile` prompt that generates a comprehensive country summary across key indicators
- FR46: The MCP server provides a `trend_analysis` prompt that guides time-series trend exploration for an indicator
- FR47: The MCP server exposes discoverable resources for popular indicators and available databases
- FR48: The MCP server exposes a workflow resource documenting the recommended 3-step data retrieval process

### Offline Performance

- NFR13: Offline indicator search must return results in under 50ms (no network calls)
- NFR14: Popular indicators and metadata files must load into memory in under 500ms at server startup
