## Issues Raised

### 1. Context Window Overflow Mid-Story

The LLM slows down and makes redundant reads when a story involves a lot of back-and-forth (edits, tests, re-reads). The context window fills up and earlier decisions get lost.

**Mitigations identified:**
- Story files should be dense and self-contained (dev log, decisions, current state all in one place so a fresh context can reload fast)
- Commit frequently (clean `git status` = fast reload)
- Shorter stories (stories like 1.5 with three tools in one were heavier, splitting reduces mid-story context bloat)

### 2. Permission Prompts Interrupting Flow

Git commands and web fetches were prompting for approval, breaking development flow unnecessarily.

**Resolution:** Configure auto-approve for low-risk commands (git, uv, curl, WebFetch).

### 3. Review Process Not Leveraging GitHub Copilot

The flow was: dev → BMAD code review. Better flow: dev → PR → GitHub Copilot review → BMAD code review using Copilot's findings as input. More data, less redundant analysis.

**Resolution (updated 2026-03-26):** Evolved during OpenClaw setup. Final flow:
- Dev implements → runs `/bmad-code-review` locally (branch diff vs main) → fixes patch findings → opens PR
- Felipe reviews PR (with Copilot assist) → leaves comments → agent addresses feedback → loop until approved
- On approval: agent merges PR and marks story done
- Rationale: running code review *before* the PR means fewer issues surface during human review, reducing round-trips.

## External Research Applied

A video case study on autonomous coding agents confirmed the same patterns:

- **Context overflow is structural, not accidental** — wrapping LLM calls inside LLM calls multiplies tokens fast
- **"Batches of 5 steps, then check" governance model** — one story = one batch, don't start the next until the previous commit is clean
- **Git as observability** — validates the PR-based review workflow
- **Scope limitation reduces hallucination** — locked stack (FastMCP, httpx, uv) + referencing CLAUDE.md and architecture docs keeps the model grounded
- **Persistent semantic memory gap** — mitigated by dense story files, `project-context.md` as quick-reload anchor, frequent commits
